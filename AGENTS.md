减少常见 LLM 编程错误的行动指南。可根据具体项目需求进行合并。

**权衡：** 这些指南偏向于谨慎而非速度。对于琐碎的任务，请自行斟酌。

## 1. 动工前先思考

**不要假设。不要隐瞒困惑。暴露出权衡取舍。**

在实现之前：

- 明确阐述你的假设。如果不确定，请提问。
- 如果存在多种理解方式，请一并呈现 —— 不要默默自行挑选。
- 如果存在更简单的方法，请直说。在有必要时予以反驳。
- 如果有任何不明确的地方，请停下来。指出困惑点。提问。

## 2. 简单至上

**用最少量的代码解决问题。不做任何投机性编码。**

- 决不提供超出需求的额外功能。
- 决不对单次使用的代码进行抽象。
- 决不引入未经要求的“灵活性”或“可配置性”。
- 决不对不可能发生的场景进行错误处理。
- 如果你写了 200 行代码，而它原本可以用 50 行搞定，重写它。

问问自己：“资深工程师会觉得这太复杂了吗？” 如果会，请简化。

## 3. 外科手术式修改

**只触碰必须触碰的部分。只清理自己留下的烂摊子。**

在修改现有代码时：

- 不要“改进”相邻的代码、注释或格式。
- 不要重构没有损坏的部分。
- 即使你有不同的习惯，也必须匹配现有的代码风格。
- 如果你注意到不相关的死代码（废弃代码），请提出来 —— 不要直接删除。

当你的修改导致部分代码孤立时：

- 移除由于你的修改而变成不再使用的导入（imports）、变量或函数。
- 除非被要求，否则不要移除此前就已存在的死代码。

检验标准：每一行修改的代码都必须能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环推进直至验证通过。**

将任务转化为可验证的目标：

- “添加校验” → “针对非法输入编写测试，然后使其通过”
- “修复 Bug” → “编写一个能复现该 Bug 的测试，然后使其通过”
- “重构 X” → “确保重构前后测试都能通过”

对于多步骤任务，简要陈述计划：

```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

清晰的成功标准能让你独立进行循环迭代。模糊的标准（如“让它跑起来”）则需要频繁的澄清确认。

**若出现以下情况，说明本指南正在发挥作用：** 代码差异（diffs）中不必要的修改减少、由于过度复杂而导致的重写减少，以及澄清问题发生在实现之前，而非在犯错之后。

## 5. 命令

- 安装依赖：`pnpm install`
- 本地开发：`pnpm dev`（仅在用户明确要求时启动）
- 打包 / 生产构建：`pnpm build`
- 启动生产服务：`pnpm start`（通常需先执行 `pnpm build`）
- 代码检查：`pnpm lint`
- 类型检查：`pnpm exec tsc --noEmit`
- Prisma 生成客户端：`pnpm dlx prisma generate`
- 数据库推送：`pnpm dlx prisma db push`
- 初始化管理员：`pnpm run init-admin`
- 填充种子数据：`pnpm run db:seed`
- 清理数据：`pnpm run db:clean`
- 导出链接：`pnpm run export:links`
- 导入链接：`pnpm run import:links -- --file ./bookmarks.json`
- 本地备份：`pnpm run simple-backup`
- 自动备份：`pnpm run auto-backup`

当前未配置 `format` 脚本，也未引入 Prettier。需要格式化时，先确认项目要采用的格式化工具与脚本，不要私自新增依赖。

## 6. 禁止

- 禁止私自引入依赖，如需可交给用户引入
- 禁止自行启动前端服务
