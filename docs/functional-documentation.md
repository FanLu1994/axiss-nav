# Axiss Nav 功能文档

本文档基于目标功能口径整理，只描述功能，不包含前端排版和视觉设计评价。

## 1. 产品定位

Axiss Nav 是一个个人链接导航与收藏管理系统，用于集中管理常用网站链接，支持链接添加、编辑、搜索、筛选、标签分类、点击统计、推荐入口、导入导出和备份。

系统应重点强化智能分析 Agent 能力，让收藏链接不只是被保存，还能被自动理解、归类、维护和持续优化。

当前系统定位为单一管理员维护的全局收藏库，不支持多用户隔离。

## 2. 管理员与认证

系统只保留一个预设好的管理员账户，不提供注册能力，也不需要用户表。

管理员账户由环境变量或部署配置预设：

- 管理员账号
- 管理员密码
- JWT 签名密钥

登录功能只校验预设管理员账号和密码。

认证方式：

- 登录成功后返回 JWT。
- 前端将 JWT 存入 `localStorage`。
- 需要权限的接口通过 `Authorization: Bearer <token>` 校验。
- JWT 有效期为 180 天。

不提供以下能力：

- 用户注册
- 用户列表
- 用户角色
- 管理员初始化页面
- 普通用户登录
- 密码找回或用户资料管理

## 3. 链接管理

链接是系统的核心数据。

每条链接包含：

- 标题
- URL
- 描述
- 图标地址
- 排序值
- 是否启用
- 点击次数
- 标签
- 分类
- 分类颜色
- 创建时间
- 更新时间

链接 URL 全局唯一。

已登录管理员可以：

- 添加链接
- 编辑链接
- 删除链接
- 重新分析链接

未登录访问者可以：

- 浏览链接
- 搜索和筛选链接
- 打开链接
- 触发点击统计

删除链接采用软删除，即将 `isActive` 设为 `false`，不会真正从数据库删除。

## 4. 添加链接

添加链接时支持两种方式：

- 手动填写标题、URL、描述、图标、标签、分类、颜色。
- 输入 URL 后点击“分析”，自动获取网站信息。

添加前会校验：

- 标题不能为空。
- URL 不能为空。
- URL 必须是 `http://` 或 `https://`。
- 活跃链接中不能存在相同 URL。

前端点击添加时，会尝试读取剪贴板。如果剪贴板内容是合法 URL，会自动填入添加表单。

## 5. URL 分析

接口：`POST /api/links/analyze`

该功能需要登录。

分析流程：

1. 校验 URL 格式。
2. 检查 URL 是否已存在。
3. 请求目标网站页面。
4. 提取网页标题。
5. 提取 favicon。
6. 如果配置了 AI 服务，则调用 AI 生成标题、描述和标签。
7. 如果 AI 不可用或失败，则使用基础信息和默认标签。

支持的 AI 环境变量包括：

- `OPENAI_API_KEY`
- `DEEPSEEK_API_KEY`
- `CLAUDE_API_KEY`
- `GEMINI_API_KEY`

当前 AI 提供商按配置顺序选择第一个可用项。

默认标签为：

- 链接
- 收藏

## 6. 智能分析 Agent 能力

智能分析 Agent 是系统的重点强化方向，用于降低管理员整理链接的成本。

Agent 应具备以下能力：

- 网页信息提取：读取网页标题、描述、关键词、favicon 和主要正文摘要。
- 内容理解：判断链接的主要用途、适用场景、工具类型和内容领域。
- 标题优化：在网页标题过长、包含站点噪声或语义不清时，生成更适合收藏展示的标题。
- 描述生成：生成简短、准确、可读的中文描述，避免空泛描述。
- 标签生成：根据链接内容生成 1 到 5 个稳定标签。
- 分类推荐：根据链接用途推荐一个主分类。
- 重复识别：识别完全重复 URL、同域名相似链接、跳转链接和可能重复的工具链接。
- 失效检测：定期检测链接是否无法访问、跳转异常或 favicon 失效。
- 批量整理：对已有链接批量补全标题、描述、图标、标签和分类。
- 维护建议：发现过多相似标签、未分类链接、低质量描述和长期未访问链接，并给出整理建议。

Agent 输出应保持可控：

- 分析结果只作为建议，不应在未确认时覆盖管理员手动编辑过的内容。
- 标签和分类应优先复用已有体系，避免为相似含义生成大量新标签。
- 描述应基于网页实际内容，不应编造网站能力。
- 分析失败时应回退到基础网页信息，不影响链接保存。

## 7. 重新分析

接口：`POST /api/links/reanalyze`

该功能需要登录。

目标功能中，重新分析应调用智能分析 Agent，重新获取网页信息并生成更新建议。

重新分析范围包括：

- 标题
- 描述
- 图标
- 标签
- 分类
- 重复或失效状态

重新分析不应直接覆盖管理员手动维护的内容，建议提供“应用全部建议”或“逐项应用”的确认机制。

## 8. 链接浏览、搜索、筛选与排序

首页会分页加载链接，每页默认 20 条，最大 `pageSize` 为 100。

支持搜索字段：

- 标题
- URL
- 描述
- 分类
- 标签

支持标签筛选：

- 根据标签文本匹配链接的 `tags` 字段。

支持分类筛选：

- 根据 `category` 精确匹配。

支持快捷筛选：

- 全部
- 常用：点击次数大于 0
- 最近添加：按创建时间倒序
- 未分类：分类为空

支持排序：

- 最新添加
- 最早添加
- 点击最多
- 标题 A-Z

页面支持滚动到底部自动加载更多。

## 9. 标签功能

标签直接存储在链接的 `tags` 字段中，格式为 JSON 字符串。

系统没有独立的标签表。

标签接口会从所有活跃链接中提取标签并统计数量，同时也会把 `category` 作为标签统计来源之一。

标签功能包括：

- 热门标签展示
- 标签点击筛选
- 标签刷新
- 根据标签名称匹配 emoji

## 10. 分类功能

分类直接存在链接的 `category` 字段中。

当前分类来源有两类：

- 添加或编辑链接时手动填写。
- URL 分析后默认取第一个标签作为分类。

首页侧边栏展示“当前分类”，来源是当前已加载链接结果中的分类，最多展示 8 个。

## 11. 点击统计

打开链接时会调用点击接口：

`POST /api/links/click`

接口会将对应链接的 `clickCount` 加 1。

点击统计用于：

- 卡片展示点击次数。
- “点击最多”排序。
- 推荐算法计算。

点击统计不要求登录。

## 12. 推荐与随机访问

推荐接口：

`GET /api/links/recommend`

当前推荐算法基于：

- 点击次数越少，分数越高。
- 添加时间越久，分数越高。
- 随机因子。

最终取分数最高的前 7 个，前端展示最多 6 个。

需要注意：当前推荐不是基于用户画像、个人历史或语义相似度，而是基于全局链接数据的简单启发式推荐。

随机访问接口：

`GET /api/links/random`

从所有活跃链接中随机选一个返回，前端点击后打开该链接并记录点击次数。

## 13. 导入导出

系统支持通过 API 导出链接。

导出接口：

- `GET /api/links/export?format=json`
- `GET /api/links/export?format=markdown`

导出需要登录。

导出内容包括：

- 标题
- URL
- 描述
- 图标
- 排序值
- 点击次数
- 创建时间
- 更新时间
- 标签
- 分类
- 颜色

支持 JSON 和 Markdown 两种格式。

导入接口：

`POST /api/links/import`

导入需要登录。

支持格式：

- JSON
- Markdown

导入时会校验：

- 标题不能为空。
- URL 不能为空。
- URL 格式正确。
- URL 不重复。

导入成功后会批量创建链接。

命令行脚本也支持导入导出：

- `pnpm run export:links`
- `pnpm run import:links -- --file ./bookmarks.json`

## 14. 备份功能

系统有三类备份能力。

本地简单备份：

`pnpm run simple-backup`

会生成：

- JSON 备份文件
- Markdown 备份文件

默认输出到 `./backups`。

GitHub 自动备份：

`pnpm run auto-backup`

需要配置：

- `GITHUB_BACKUP_REPO`
- `GITHUB_TOKEN`
- 可选 `BACKUP_BRANCH`

它会生成 Markdown 备份，并推送到 GitHub 仓库指定分支。

Vercel 定时备份：

`vercel.json` 配置了 cron：

```json
{
  "path": "/api/cron/backup",
  "schedule": "0 0 * * *"
}
```

对应接口会每天 0 点执行 GitHub 备份。

需要注意：cron 路由位于 `api/cron/backup.js`，使用 GitHub REST API 上传 Markdown 备份和更新 README。

## 15. 数据模型

当前核心数据模型只需要链接 `Link`。

系统不需要用户表。管理员认证信息由配置提供，不落库保存为业务数据。

链接 `Link`：

- id
- title
- url
- description
- icon
- order
- isActive
- clickCount
- tags
- category
- color
- analysisStatus
- lastAnalyzedAt
- lastCheckedAt
- isAvailable
- createdAt
- updatedAt

链接不关联用户。

可选的结构化标签模型 `Tag`：

- id
- name
- slug
- color
- icon
- count
- createdAt
- updatedAt

可选的链接标签关联模型 `LinkTag`：

- linkId
- tagId

## 16. 权限边界

公开可用：

- 查看链接
- 搜索链接
- 筛选链接
- 获取推荐链接
- 获取随机链接
- 查看热门标签
- 打开链接并记录点击

需要管理员登录：

- 添加链接
- 编辑链接
- 删除链接
- 分析 URL
- 重新分析链接
- 导入链接
- 导出链接

不提供：

- 注册账户
- 初始化管理员
- 用户管理
- 用户角色管理

## 改良建议

1. 落实单管理员认证

   移除注册接口、初始化管理员流程和用户表，改为通过环境变量校验唯一管理员账户，减少不必要的数据模型和权限复杂度。

2. 修正“重新分析”功能

   重新分析应复用智能分析 Agent，重新抓取网页并生成标题、描述、图标、标签、分类、失效状态和重复风险建议。

3. 标签应结构化

   当前标签存在 JSON 字符串里，搜索和统计都依赖字符串匹配。建议后续改为独立 `Tag` 表和关联表，方便去重、统计、改名、合并和精确筛选。

4. 强化智能分析 Agent

   将 Agent 作为核心能力建设，而不是只在添加链接时做一次 AI 描述生成。它应支持单条分析、重新分析、批量分析、失效检测、重复识别和整理建议。

5. 保护手动编辑内容

   管理员手动修改过的标题、描述、标签和分类应有更高优先级。Agent 重新分析时只生成建议，除非管理员确认，否则不自动覆盖。

6. 增加分析状态

   链接应记录 `analysisStatus`、`lastAnalyzedAt`、`lastCheckedAt`、`isAvailable` 等字段，用于区分未分析、分析中、分析失败、已分析、链接失效等状态。

7. 标签治理

   提供标签合并、改名、删除、批量替换能力。Agent 可以建议合并近义标签，例如“AI 工具”“人工智能工具”“AI应用”。

8. 分类体系治理

   分类应保持少量、稳定、互斥。Agent 生成分类时应优先从已有分类中选择，只有确实不匹配时才建议新分类。

9. 强化安全配置

   `JWT_SECRET` 当前有 fallback 默认值，生产环境存在风险。建议在启动或接口调用时强制要求配置 `JWT_SECRET`。

10. 导入去重逻辑优化

当前导入检查 URL 时会把软删除链接也视为重复。建议明确策略：恢复软删除链接，还是允许重新导入。

11. 推荐算法命名更准确

当前推荐更像“低点击 + 老链接 + 随机曝光”。建议前端文案改成“发现链接”或“轮换入口”，避免用户误解为智能个性化推荐。

12. 增加操作审计或备份触发入口

目前备份主要靠脚本和 cron。可以增加管理端手动备份入口，并展示最近一次备份状态。

13. 增加批量管理

链接变多后，建议支持批量删除、批量改分类、批量打标签、批量导入预览。

14. 增加批量分析入口

管理员可以选择未分析、未分类、描述为空、图标为空、长期未检查的链接，批量交给 Agent 分析并生成待确认建议。

15. 增加链接健康检查

定期检查链接可访问性，标记无法访问、重定向异常、证书错误或域名失效的链接，便于管理员清理收藏库。
