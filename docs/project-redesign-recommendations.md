# Axiss Nav 改造建议

## 目标

把 Axiss Nav 从“能用的个人收藏夹”改造成“高频使用的个人导航工作台”。改造重点不是拆成前后端两个项目，而是在现有 Next.js 全栈单体内完成产品、交互、视觉和代码组织升级。

成功标准：

1. 首页能让用户快速找到、添加、管理和重新整理链接。
2. 新增链接流程清晰，不依赖用户猜测下一步。
3. 搜索、筛选、推荐、标签、导入导出都有明确入口和状态反馈。
4. 视觉风格统一，减少漂浮按钮、过度装饰、默认卡片和不一致的圆角/阴影。
5. API Route 保持轻薄，业务逻辑沉到服务层，后续功能扩展不再堆在页面和 route 文件里。

## 当前判断

当前项目适合继续采用 Next.js 全栈单体，而不是立即前后端分离。

原因：

- 功能边界集中在个人导航、认证、链接管理、AI 分析、推荐、导入导出和备份。
- 数据模型简单，核心是用户和链接。
- 前端通过同源 `/api/*` 调用后端，没有明显跨端 API 复用需求。
- 独立后端会引入 CORS、跨域鉴权、部署拆分、接口契约维护等成本，但短期收益有限。

建议方向：

- 不做物理拆分。
- 做“内部模块化”：页面层、组件层、API 层、服务层、数据层职责分清。
- 优先解决功能完整度和体验问题，再做架构清理。

## 主要问题

### 功能问题

1. 链接管理能力偏弱

   当前主要支持新增、删除、点击、重新分析。缺少编辑、批量管理、排序、归档恢复、重复链接处理、失效检测等常见管理能力。

2. 标签体系不完整

   标签以 JSON 字符串存在 `Link.tags` 中，短期可用，但不利于统计、重命名、合并、颜色管理和精确筛选。

3. 搜索和筛选体验粗糙

   当前搜索依赖回车触发，标签点击会改变搜索词，但缺少显式筛选状态、清除筛选、组合筛选、排序切换。

4. 推荐逻辑偏随机

   推荐基于点击次数、创建时间和随机因子，缺少用户可理解的解释，也缺少“常用”“最近添加”“最近访问”“长期未访问”等更实用的入口。

5. 添加链接流程不完整

   当前流程是输入 URL、分析、确认添加。缺少手动编辑分析结果、跳过 AI、分析失败后的可恢复路径、重复链接跳转、批量粘贴。

6. 登录和初始化体验割裂

   登录页、初始化页和首页视觉语言不一致。初始化页还有“创建后将跳转到登录页面”的提示，但实际代码会保存 token 并跳转主页，文案和行为不一致。

7. 缺少管理视图

   首页同时承担浏览、搜索、推荐、添加、删除、重新分析等职责。随着功能增加，应该拆出管理模式或管理页。

### 交互问题

1. 首页层级不清晰

   搜索、推荐、随机标签、链接列表、登录状态、主题切换、添加按钮分散在页面各处。用户第一次进入时不知道主操作是什么。

2. 右键菜单不适合移动端

   右键菜单适合桌面，但移动端没有对应的长按或更多按钮入口。

3. 添加入口不够明确

   加号按钮只有登录后出现，且在搜索框旁边。它是高频操作，应该在桌面和移动端都有稳定、可发现的入口。

4. 状态反馈过度依赖 toast

   表单错误、分析失败、重复链接等更适合在当前流程内显示，而不是全部弹 toast。

5. 空状态不够有用

   “暂无收藏网址”不能指导用户下一步。空状态应该直接提供添加链接、导入书签、查看示例的入口。

### 视觉问题

1. 风格不统一

   首页有粒子背景、玻璃拟态、蓝色渐变按钮、圆形按钮、虚线搜索动效；登录页则是默认居中卡片。整体像多个时期叠加出来的界面。

2. 装饰多于信息

   粒子背景和透明卡片弱化了链接内容。导航工具应优先保证可读性、扫描效率和操作稳定。

3. 卡片密度不稳定

   链接卡片、推荐区域、随机标签缺少统一的信息结构。用户难以快速比较标题、域名、标签和访问频率。

4. 移动端操作需要重做

   当前布局更偏桌面。移动端应优先考虑搜索、添加、筛选、卡片操作按钮的拇指可达性。

## 产品改造建议

### 1. 首页重构为导航工作台

建议首页分为四个区域：

1. 顶部工具栏

   包含产品名、搜索框、添加按钮、导入入口、用户菜单、主题切换。

2. 快捷筛选区

   包含全部、常用、最近添加、最近访问、未分类、失效链接、标签筛选。

3. 主链接区

   默认显示全部链接，支持网格视图和紧凑列表视图。

4. 侧边信息区或折叠面板

   桌面端显示推荐、标签统计、最近操作；移动端收进抽屉或底部面板。

验收标准：

- 用户进入首页后，3 秒内能看懂“搜索、添加、筛选、管理”四个主操作。
- 添加链接入口在桌面和移动端都稳定可见。
- 推荐不再压在链接列表上方抢主视觉，而是作为辅助区域。

### 2. 添加链接流程升级

建议流程：

1. 输入或粘贴 URL。
2. 系统检查重复和 URL 有效性。
3. 并行获取标题、图标、描述和 AI 标签。
4. 用户可编辑标题、描述、标签、分类。
5. 保存后在当前列表中插入，并显示明确成功状态。

必须补齐的状态：

- URL 无效。
- 链接已存在。
- 网页信息获取失败。
- AI 不可用。
- AI 分析超时。
- 保存失败。

验收标准：

- AI 不可用时，用户仍然可以手动添加链接。
- 分析结果允许用户修改。
- 重复链接提示能直接定位到已有链接。

### 3. 链接管理补全

优先级从高到低：

1. 编辑链接

   支持修改标题、URL、描述、标签、分类、图标。

2. 软删除后的恢复

   既然现在是 `isActive = false`，就应该提供回收站或恢复入口。

3. 批量操作

   支持多选后批量删除、批量加标签、批量改分类、批量重新分析。

4. 手动排序

   支持置顶、拖拽排序，或至少支持“按创建时间、点击次数、标题、最近访问”排序。

5. 失效检测

   定期或手动检测链接是否可访问，显示失效状态。

验收标准：

- 单个链接不需要删除重建即可修改。
- 用户能批量整理从浏览器导入的大量链接。
- 删除操作可恢复，降低误操作风险。

### 4. 搜索和筛选升级

建议能力：

- 输入时防抖搜索，不只依赖回车。
- 显示当前筛选条件，例如“标签：设计”“分类：工具”。
- 支持一键清除筛选。
- 支持排序切换。
- 支持只看未分类、只看失效、只看最近添加、只看高频访问。

短期可以继续使用 PostgreSQL `contains` 查询。数据量变大后再考虑全文搜索或专门搜索索引。

验收标准：

- 搜索、标签筛选、排序状态在 UI 上清晰可见。
- 用户能在不刷新页面的情况下连续调整筛选条件。

### 5. 标签和分类重构

短期建议：

- 保留 `Link.tags` JSON 字符串，先在 UI 上补齐标签管理。
- 增加标签重命名、合并、删除、颜色选择。
- 增加“未分类”视图。

中期建议：

- 把标签拆成独立表：`Tag`、`LinkTag`。
- 分类如果仍然是一对一，可以保留在 Link 上；如果要多级分类，再建 `Category` 表。

验收标准：

- 用户能整理 AI 生成的不准确标签。
- 标签列表能展示数量，并支持点击筛选。
- 标签重命名后，所有相关链接同步更新。

### 6. 推荐和快捷入口重做

把“推荐”改成更可解释的快捷集合：

- 常用：点击次数高。
- 最近添加：创建时间新。
- 最近访问：需要新增 `lastClickedAt`。
- 冷门收藏：长期未点击但可能值得回顾。
- 随机发现：保留随机，但明确为探索入口。

验收标准：

- 每个推荐区都有明确原因，不只是随机排序。
- 用户可以关闭不需要的推荐区。

### 7. 导入导出和备份产品化

当前已有脚本和 API，但入口不够产品化。

建议增加“数据管理”页面：

- 导入浏览器书签 HTML 或 JSON。
- 导入前预览数量、重复数量、即将新增数量。
- 导入后显示成功、跳过、失败明细。
- 导出 JSON、Markdown、浏览器书签格式。
- 显示最近备份时间和备份状态。

验收标准：

- 用户不需要命令行也能完成导入导出。
- 大批量导入失败时能知道哪几条失败。

## 视觉改造建议

### 设计方向

建议定位为“安静、清晰、高频使用的个人工作台”，不要做成营销页或强装饰应用。

视觉原则：

- 背景以中性浅色和深色为主。
- 只保留一个主强调色。
- 减少渐变、粒子、强玻璃效果。
- 卡片用于链接项，不把每个页面区域都做成卡片。
- 图标、标签、按钮尺寸统一。
- 移动端优先保证操作可达性。

### 首页建议布局

桌面端：

- 左侧或顶部为主导航。
- 中间为链接列表。
- 右侧为标签、推荐和数据概览。

移动端：

- 顶部搜索。
- 底部固定操作栏：添加、筛选、管理、我的。
- 链接卡片改为单列紧凑布局。

### 链接卡片建议

每张卡片包含：

- favicon 或域名首字母。
- 标题。
- 域名。
- 描述，最多两行。
- 标签，最多显示 3 个，剩余折叠。
- 操作按钮：打开、编辑、更多。

交互：

- 点击卡片主体打开链接。
- 桌面端 hover 显示次级操作。
- 移动端始终显示更多按钮，不依赖右键。

### 表单和弹窗建议

- 标签必须在输入框上方。
- 错误信息显示在输入框下方。
- 主要按钮只放一个。
- 加载状态要说明当前正在做什么。
- 分析链接这种长任务应显示阶段状态：检查链接、获取网页信息、生成标签、等待确认。

## 技术改造建议

### 1. 服务层拆分

建议新增目录：

```text
src/server/auth
src/server/links
src/server/tags
src/server/import-export
src/server/ai
```

API Route 只保留：

- 读取 request。
- 调用鉴权。
- 调用服务函数。
- 返回 response。

示例目标：

```text
src/app/api/links/route.ts
src/server/links/link-service.ts
src/server/links/link-repository.ts
src/server/links/link-types.ts
```

验收标准：

- 单个 API Route 尽量控制在 80 行以内。
- 业务逻辑可以被 API、脚本和未来任务复用。

### 2. 前端状态整理

首页当前承担太多状态。建议拆分：

```text
src/features/home/home-page.tsx
src/features/links/link-grid.tsx
src/features/links/link-card.tsx
src/features/links/add-link-dialog.tsx
src/features/filters/filter-bar.tsx
src/features/user/user-menu.tsx
```

短期仍使用本地 state，不急于引入全局状态库。

验收标准：

- `src/app/page.tsx` 只负责组装页面。
- 添加、筛选、列表、用户菜单有清晰边界。

### 3. API 类型和校验

建议为请求和响应建立明确类型：

```text
src/shared/types/link.ts
src/shared/types/api.ts
```

如果需要运行时校验，再考虑引入校验库。当前项目规则禁止私自引入依赖，所以这一步应先用简单函数完成，等确认后再引入新依赖。

验收标准：

- 前端不再重复手写多个不一致的 Link interface。
- API 返回结构统一包含 `data` 或 `error`。

### 4. 认证体验和安全

当前 token 存在 `localStorage`。短期可继续使用，但要统一处理过期和无效 token。中期建议迁移到 HttpOnly Cookie，减少 XSS 后 token 泄漏风险。

建议：

- 增加统一 `fetcher`，自动附带 token。
- 401 时统一退出登录或跳转登录。
- 登录页补充 label、错误文案和加载状态。
- 初始化页修正文案和实际跳转不一致的问题。

验收标准：

- 所有需要登录的操作在 token 失效时表现一致。
- 登录、初始化、首页用户状态一致。

### 5. 性能和数据

建议：

- 链接列表继续分页，但增加排序参数。
- 推荐接口不要每次全量读取所有链接后在内存排序。
- 增加 `lastClickedAt` 字段。
- 点击统计和最近访问分开存储。
- JSON tags 在中期改成关系表。

验收标准：

- 链接数量达到几千时，首页仍能稳定加载。
- 推荐接口查询量和计算量可控。

## 分阶段实施计划

### 第一阶段：体验骨架重做

范围：

- 重构首页布局。
- 明确顶部工具栏、筛选区、链接区、辅助区。
- 统一登录页、初始化页、首页视觉风格。
- 移除或弱化粒子背景和过度装饰。
- 为移动端增加稳定操作入口。

验证：

- `pnpm lint`
- `pnpm exec tsc --noEmit`
- 手动检查桌面和移动端主要页面。

### 第二阶段：添加和管理闭环

范围：

- 添加链接支持编辑分析结果。
- AI 不可用时允许手动添加。
- 增加编辑链接。
- 增加恢复或回收站。
- 链接卡片增加更多菜单，替代仅依赖右键。

验证：

- 未登录不能新增、编辑、删除。
- 已登录可以新增、编辑、删除、恢复。
- 重复链接、分析失败、保存失败都有明确状态。

### 第三阶段：搜索、筛选和推荐

范围：

- 搜索改为防抖触发。
- 增加筛选状态展示和清除。
- 增加排序。
- 推荐拆成常用、最近添加、最近访问、随机发现。
- 新增 `lastClickedAt`。

验证：

- 搜索、标签、分类、排序组合后结果正确。
- 点击链接后最近访问能更新。
- 推荐区刷新后原因可解释。

### 第四阶段：数据管理

范围：

- 做数据管理页。
- 产品化导入、导出、备份状态。
- 导入前预览重复项和失败项。

验证：

- 可从 UI 完成导入导出。
- 导入重复链接不会破坏现有数据。
- 导出文件可再次导入。

### 第五阶段：内部架构整理

范围：

- 抽离 `src/server/*` 服务层。
- 整理共享类型。
- 收敛 API 返回格式。
- 拆分首页大组件。

验证：

- 功能行为不变。
- API Route 明显变薄。
- 页面组件职责清晰。

## 不建议现在做的事

1. 不建议立即前后端分离。

   当前收益不足，复杂度会上升。

2. 不建议立即引入大型状态管理库。

   先把组件边界拆清楚，本地状态足够支撑近期改造。

3. 不建议先做标签表大迁移。

   先补齐标签管理体验，再根据真实需求迁移数据结构。

4. 不建议继续堆视觉特效。

   当前主要问题是信息结构和操作效率，不是缺动画。

5. 不建议私自引入格式化工具或新依赖。

   项目说明里已有约束。需要新增依赖时，应先确认。

## 推荐优先级

最高优先级：

1. 首页布局和移动端操作入口。
2. 添加链接流程。
3. 编辑链接。
4. 搜索筛选状态。
5. 视觉统一。

中优先级：

1. 回收站。
2. 批量操作。
3. 数据管理页。
4. 推荐逻辑重做。

低优先级：

1. 标签关系表迁移。
2. HttpOnly Cookie 认证迁移。
3. 独立后端。

## 建议的第一版改造范围

如果只做一轮改造，建议控制在以下范围：

1. 重做首页布局。
2. 重做链接卡片。
3. 重做添加链接弹窗。
4. 增加编辑链接。
5. 增加筛选栏和排序。
6. 统一登录页和初始化页视觉。

这轮完成后，项目会从“功能堆叠”变成“可持续使用的个人导航工具”。后续再做导入导出产品化、批量管理和数据模型升级会更稳。
